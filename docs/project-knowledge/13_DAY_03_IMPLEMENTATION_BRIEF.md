# Day 3 Implementation Brief: Secure Scanner Import Foundation and Nmap XML

## Objective

Build the first scanner-import vertical slice. An authorised Admin or assigned Consultant can import an **Nmap XML report** into an Assessment through a Django management command. SARP safely parses the report, creates or matches Assets, creates deduplicated ScannerObservations, and preserves import metadata.

At the end of Day 3, Nmap output is **evidence awaiting review**, not an automatic vulnerability Finding. There is no React upload screen, no ZAP/Nessus/Burp parser, no triage UI, no AI/RAG, and no Azure work today.

## Product boundary

SARP imports reports produced by authorised scanning work performed outside SARP. It never accepts a hostname/IP/URL and launches Nmap itself.

The only manual import command required today is:

```bash
python manage.py import_scan \
  --assessment-id 1 \
  --tool nmap \
  --file fixtures/nmap/sample.xml \
  --actor-email consultant@sarp.local
```

`--actor-email` is required. A management command has no logged-in request user, so it must resolve a real SARP user and apply the same tenant-write rules as an API endpoint.

## Data model

### 1. `ScanImport`

Represents one processed report file. It stores metadata and results, never the raw report content.

```text
assessment                → Assessment
source_tool               → NMAP today; choices prepared for ZAP, NESSUS, BURP
source_filename           → original filename only
file_sha256               → 64-character SHA-256 checksum
imported_by               → accounts.User
status                    → PROCESSING | COMPLETED | FAILED
created_at
completed_at              → nullable
observations_created      → integer, default 0
observations_updated      → integer, default 0
error_summary             → safe, short, nullable text
```

### 2. `ScannerObservation`

Represents a canonical scanner observation within an Assessment. It is not yet a confirmed security Finding.

```text
assessment                → Assessment
asset                     → Asset, nullable until matched
source_tool               → NMAP today
external_id               → stable ID generated by importer
scanner_plugin_id         → e.g. nmap-port or NSE script ID; nullable
fingerprint               → stable SHA-256/hex string
triage_status             → NEW today; full workflow comes Day 5
title
description
evidence_summary          → sanitised and length-limited
raw_severity              → nullable
confidence                → nullable
port                      → nullable integer
protocol                  → nullable string
url                       → nullable
suggested_remediation     → nullable
cve_ids                   → JSON list, default []
cwe_ids                   → JSON list, default []
references                → JSON list, default []
raw_location              → nullable, safe text
first_seen_at
last_seen_at
first_seen_import         → ScanImport, nullable
last_seen_import          → ScanImport, nullable
created_at
updated_at
```

Add a uniqueness rule:

```text
unique(assessment, fingerprint)
```

This is the Day 3 deduplication guarantee. Re-importing the same Nmap evidence updates `last_seen_at` and `last_seen_import`; it must not create a duplicate `ScannerObservation`.

### 3. `ScanImportObservation`

This small join model preserves history: it records that a canonical observation appeared in a particular ScanImport.

```text
scan_import               → ScanImport
scanner_observation       → ScannerObservation
state                     → CREATED | REOBSERVED
observed_at
unique(scan_import, scanner_observation)
```

Without this model, re-importing the same observation would overwrite its connection to the earlier report. This is needed for a future `GET /imports/{id}/observations/` endpoint.

### 4. `FindingSource`

Create the schema now for traceability, but do not implement promotion logic today.

```text
finding                   → Finding
scanner_observation       → ScannerObservation
first_seen_at
last_seen_at
unique(finding, scanner_observation)
```

## Parser contract

Create a neutral dataclass in `apps/imports/parsers/base.py`:

```python
from dataclasses import dataclass, field


@dataclass(frozen=True)
class NormalisedObservation:
    source_tool: str
    external_id: str
    title: str
    raw_severity: str | None = None
    confidence: str | None = None
    asset_identifier: str | None = None
    hostname: str | None = None
    port: int | None = None
    protocol: str | None = None
    url: str | None = None
    description: str = ""
    evidence_summary: str = ""
    suggested_remediation: str | None = None
    cve_ids: list[str] = field(default_factory=list)
    cwe_ids: list[str] = field(default_factory=list)
    scanner_plugin_id: str | None = None
    references: list[str] = field(default_factory=list)
    raw_location: str | None = None
```

Create `BaseImporter`:

```python
from abc import ABC, abstractmethod


class BaseImporter(ABC):
    source_tool: str

    @abstractmethod
    def validate(self, content: bytes, filename: str) -> None:
        """Raise a safe ImportValidationError for bad or unsupported input."""

    @abstractmethod
    def parse(self, content: bytes) -> list[NormalisedObservation]:
        """Return normalised values only. Never write to the database here."""
```

### Non-negotiable design rule

Parser classes only parse bytes into `NormalisedObservation` objects. They do **not** create Models, check memberships, or write audit data. The service layer owns database work and permissions.

## Nmap XML importer

Implement `NmapXmlImporter` in `apps/imports/parsers/nmap.py`.

### Security requirements

- Add `defusedxml` to backend requirements.
- Use `defusedxml.ElementTree`, never the standard-library XML parser directly.
- Reject XML with a `DOCTYPE`, entity expansion, malformed structure, unsupported file extension, blank content, or a file over a bounded local limit.
- For Day 3, use a maximum report size of **5 MiB** configurable through a Django setting such as `MAX_IMPORT_FILE_SIZE_BYTES`.
- Convert parser failures into a project-specific `ImportValidationError` with a safe message. Do not return XML internals, stack traces, local file paths, secrets, or raw report content.
- Never save the XML bytes to `ScanImport`, `ScannerObservation`, logs, error messages, or audit metadata.

### Nmap data mapping

For each host with a usable address or hostname:

```text
Host IPv4/IPv6 address or hostname → Asset candidate
Open port + protocol + service      → port exposure observation
NSE script ID + output              → script observation
```

#### Port exposure observation

For each `state="open"` port, create a normalised observation similar to:

```text
external_id: host:192.0.2.10:tcp:443:port
scanner_plugin_id: nmap-port
fingerprint inputs: nmap | host identifier | tcp | 443 | nmap-port
title: Open TCP/443 (https)
description: Nmap observed an open TCP port 443 with the https service.
evidence_summary: state=open; service=https; product=...; version=...
raw_location: 192.0.2.10:443/tcp
```

An open port is an **observation**, not automatically a vulnerability or Finding.

#### NSE script observation

For each script result under an open port, create a separate observation:

```text
external_id: host:192.0.2.10:tcp:443:script:ssl-enum-ciphers
scanner_plugin_id: ssl-enum-ciphers
fingerprint inputs: nmap | host identifier | tcp | 443 | ssl-enum-ciphers
title: Nmap NSE: ssl-enum-ciphers
description: Nmap NSE script ssl-enum-ciphers returned output for TCP/443.
evidence_summary: sanitised, compact script output
raw_location: 192.0.2.10:443/tcp
```

### Evidence handling

Implement a small pure helper that:

- Removes control characters.
- Collapses excessive whitespace.
- Limits evidence to a documented maximum, e.g. 2,000 characters.
- Appends a clear truncation marker when needed.

This is not a promise of perfect secret detection. It is a Day 3 guardrail against storing raw scanner reports and excessively large output.

## Import service

Create a service such as:

```python
def import_report(
    *,
    assessment: Assessment,
    actor: User,
    tool: str,
    filename: str,
    content: bytes,
) -> ScanImport:
    ...
```

The service must:

1. Verify the actor can write records for `assessment.client` using `can_write_client_records` from `apps.tenancy.selectors`.
2. Select the parser from a small registry. Only `nmap` is enabled today.
3. Validate and parse the input.
4. Calculate the file SHA-256.
5. Create `ScanImport` with `PROCESSING` status.
6. For each normalised observation:
   - Match or create an `Asset` in the same Assessment.
   - Compute a deterministic fingerprint.
   - Create a new `ScannerObservation`, or update the existing matching one.
   - Create `ScanImportObservation` with `CREATED` or `REOBSERVED` state.
7. Update created/updated counters.
8. Mark the `ScanImport` as `COMPLETED` and record `completed_at`.
9. On validation failure, create no partial records. Return/raise a safe domain error.
10. Do not create a `Finding` today.

### Asset matching rules for Nmap

Within the selected Assessment, match in this order:

1. Exact IP address match.
2. Exact hostname match if there is no IP match.
3. Create a `HOST` Asset if no match exists.

Use a safe display name such as hostname first, then IP address. Do not create a duplicate Asset when the same Nmap host appears again.

## Test plan

Use fictional fixtures only. Add focused tests under `apps/imports/tests/`.

### Parser tests

- Valid Nmap fixture parses one host, expected open-port observations, and at least one NSE script observation.
- An open port has `nmap-port` as the plugin ID and does not become a `Finding`.
- NSE observation uses the script ID as `scanner_plugin_id`.
- Blank content, wrong extension and malformed XML raise `ImportValidationError` with safe messages.
- XML containing a forbidden entity/DOCTYPE is rejected.
- Evidence output is sanitised and truncated.
- Parser methods never write database records.

### Service and model tests

- Import creates one `ScanImport` with filename and SHA-256 but no raw XML content field.
- Import creates/matches Assets within the chosen Assessment.
- Import creates `ScannerObservation` and `ScanImportObservation` records.
- Re-importing the same fixture creates a new `ScanImport`, creates no duplicate canonical observations or Assets, updates `last_seen_at`, and creates a second `ScanImportObservation` link.
- Nmap import does not create a `Finding`.
- `FindingSource` has its uniqueness constraint.

### Permission/API tests

Create Admin, Consultant A assigned to Client A, Consultant B assigned to Client B, Manager A, and Client User A.

- Consultant A can import into Client A's Assessment.
- Consultant B cannot import into Client A's Assessment.
- Manager and Client User cannot run a write import through the service/command.
- Consultant B cannot list/retrieve Client A imports or observations.
- Manager A can read assigned Client A imports/observations.
- Client User A cannot read raw imports/observations.
- Unauthenticated API requests are denied.

### Management command tests

- Valid command imports the fictional fixture.
- Unknown actor email fails safely.
- Unknown assessment ID fails safely.
- Unsupported `--tool` fails safely.
